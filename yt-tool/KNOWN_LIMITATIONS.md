# Known Limitations

本文档记录当前版本的已知限制与边界行为，便于后续开发、测试和使用时统一认知。

---

## 1. Playlist URL 当前仅探测第一个条目

### 现状
当输入为 playlist URL 时，当前探测逻辑只会读取播放列表的第一个条目用于格式分析。

### 影响
- 菜单中显示的格式信息只对应首条视频
- 不代表当前版本已经完整支持整个播放列表下载流程
- 用户可能误以为"播放列表已完整支持"

### 后续建议
后续可扩展为：

- 明确提示"检测到 playlist"
- 让用户选择：
  - 仅处理首条
  - 处理整个播放列表
- 整个播放列表下载时按 `%(playlist_title)s` 建目录隔离

---

## 2. 自动字幕尚未支持下载

### 现状
当前探测结果会区分：

- 普通字幕（`subtitles`）
- 自动字幕（`automatic_captions`）

但下载逻辑目前只处理普通字幕，不处理自动字幕。

### 影响
- 某些视频虽然存在自动字幕，但当前版本中无法通过程序直接下载
- 用户可能看到"另有 N 个自动字幕"，但菜单里无法实际选择这些轨道

### 后续建议
后续可扩展为：

- 在菜单中区分"普通字幕"和"自动字幕"
- 当普通字幕不存在时，允许用户选择自动字幕
- 为自动字幕单独增加下载函数或增加下载参数分支

---

## 3. 某些"字幕"轨道实际可能是 live_chat

### 现状
对于直播回放等视频，yt-dlp 返回的某些字幕轨道可能不是常规字幕，而是 `live_chat`。

### 表现
下载后得到的文件可能为：

- `.live_chat.json`
- 或其他 JSON 相关输出

而不是常规的：

- `.srt`
- `.vtt`

### 影响
- 从程序角度看，这不是错误
- 但从普通用户视角看，容易误以为"字幕下载不正常"

### 后续建议
- 在菜单中更明确标记 `live_chat`
- 下载前增加提示：该轨道不是常规字幕，下载结果可能为 JSON
- 后续可考虑把 `live_chat` 与常规字幕分开展示

---

## 4. Windows 启动器尚需真实 Windows 环境最终验收

### 现状
当前核心 Python 流程已经完成联调与自动化测试，但 Windows 启动器的最终行为仍依赖：

- `cmd.exe`
- Windows PowerShell
- Windows 上的 PATH 与 Python 安装方式
- Windows 下的路径规则与执行策略

### 影响
- 在 macOS 上无法真实验证 `.cmd`
- 在 macOS 上可部分验证 `.ps1` 逻辑，但不能完全替代 Windows 实机测试

### 后续建议
在真实 Windows 环境中至少完成以下验证：

- 双击 `yt.cmd`
- PowerShell 运行 `yt.ps1`
- 带 URL 参数启动
- 不带参数进入交互
- Python 在 PATH 中 / 不在 PATH 中 两种情况

---

## 5. 当前没有"最终文件路径摘要"输出

### 现状
下载成功后，程序会打印"XX下载完成"以及部分 `yt-dlp` 输出，但没有统一汇总：

- 最终文件路径
- 最终文件名
- 下载类型
- 字幕类型说明

### 影响
- 用户需要自行查看 `yt-dlp` 输出或目标目录
- 当输出较长时，不够直观

### 后续建议
增加统一结果摘要，例如：

- 下载类型
- 保存目录
- 文件名
- 是否为 `live_chat`
- 是否为合并输出

---

## 6. 当前缺少格式默认推荐/排序优化

### 现状
程序当前会列出可用流，但没有基于偏好项给出默认推荐。

虽然配置中已经定义了偏好，例如：

- 视频偏好高度
- 视频编码偏好
- 视频容器偏好
- 音频编码偏好

但这些偏好目前尚未完整用于菜单推荐或排序优化。

### 影响
- 用户需要自行判断选哪个流更合适
- 可用性还有提升空间

### 后续建议
后续可加入：

- 默认推荐最佳视频流
- 默认推荐最佳音频流
- 按用户偏好排序显示

---

## 总结

当前版本已达到"核心流程可用、联调完成、自动化测试覆盖到位"的状态，但以下能力仍建议后续继续完善：

1. 完整 playlist 支持
2. 自动字幕支持
3. `live_chat` 更清晰的提示与展示
4. Windows 启动器真机验收
5. 下载结果摘要优化
6. 格式推荐与排序优化
