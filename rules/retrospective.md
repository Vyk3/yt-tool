# 复盘模板

中高风险任务与交付节点任务（PR merge、release 发布、CI 修复收口、直接在 main 推进）在任务关闭前完成。

以下命令异常也强制触发最小复盘：`push` / `pr create` / `pr merge`（退出码非 0 或出现异常输出）。

## 最小产物

**事实时间线**：发生了什么（按时间顺序列出关键事件）

**根因**：
- 流程缺口（哪个检查节点缺失）
- 技术缺口（哪个假设不成立）

**临时止血动作**：本次采取了什么来恢复现状

**永久防护**：将转化为哪条可复用规则或自动化检查

**验证证据**：如何确认不再复发

## 错误来源覆盖范围

复盘默认覆盖以下来源，不限于代码缺陷本身：

- **会话内错误**：误判需求、遗漏约束、重复返工、错误命令选择、错误工具选择
- **CI / 发布失败**：本地未发现的问题、脚本假设不成立、环境差异、打包链路问题
- **仓库历史返工**：同类问题反复修补、一次修复后在相邻任务再次出现
- **误用工具 / 命令**：本可脚本化的步骤被临场发挥替代，导致输出漂移或执行偏差
- **输出噪音导致的误判**：结论被大段过程淹没，主会话未能及时识别阻断项或证据

复盘关注的是“失败模式”，而不是单次措辞或临时上下文细节。

---

## 规则提炼准入（满足全部三条才可写入 AGENTS.md）

- **可泛化**：适用于同类任务，而非一次性细节
- **可验证**：可通过检查、脚本或流程节点验证执行
- **有收益**：可降低复发概率或定位成本

不满足准入的结论保留在 PR/issue 复盘，不进入长期原则文档。

每条新增规则须标注：
- 对应失败模式（防止什么问题）
- 执行锚点（在哪个阶段检查）

---

## 复盘产物去向判定

复盘完成后，必须为每条结论选择一个明确去向，不保留“以后再看”的悬空结论。

### 写入 `rules/`

满足以下任一特征时，进入规则文档：

- 本质是**约束**、**判定标准**、**触发条件**、**准入条件**
- 目标是减少自由度，而不是描述一套具体操作步骤
- 可在任务开始前、提交前、评审前等固定阶段执行检查

典型例子：

- 哪类改动必须触发独立 Reviewer
- 哪类命令必须单独确认
- 哪类复盘结论不允许直接升格为全局原则

### 写入 `skills/`

满足以下条件时，进入 skill / workflow 文档：

- 场景高频复发
- 存在稳定的执行顺序
- 多步流程比单条规则更重要
- 执行者需要“照流程做”而不只是“记住约束”

典型例子：

- CI 失败归因流程
- 发布前检查清单
- 高风险 review / re-review 流程

### 写入脚本或命令包装

满足以下条件时，优先收敛为脚本、固定命令模板或稳定入口，而不是继续靠人记忆：

- 重复解释成本高
- 命令参数边界清晰
- 同类任务反复出现执行偏差
- 自动化入口能显著降低误用概率

典型例子：

- 统一 CI 查询入口
- 固定格式整理命令
- 固定验证命令模板

### 仅保留在复盘记录

出现以下情况时，不进入长期资产：

- 只适用于单一任务、单一上下文
- 不可泛化或不可验证
- 只是临时提示词细节，没有形成稳定失败模式
- 成本高于收益

默认不把偶发的会话措辞问题写进全局规则，除非它已经导致重复返工或跨任务复发。

---

## 新增长期资产的最小元数据

无论进入 `rules/`、`skills/` 还是脚本入口，每条沉淀结论至少标注以下信息：

- **失败模式**：防止什么问题再次发生
- **适用范围**：哪些任务、哪些目录、哪些运行时适用
- **执行锚点**：在哪个阶段检查或执行
- **验证方式**：如何确认该沉淀真正起效
- **跨运行时通用性**：是 Claude / Codex 共用，还是仅适用于某一运行时
