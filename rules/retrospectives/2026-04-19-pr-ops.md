# 复盘记录：PR 操作异常（2026-04-19）

## 范围

- 仓库：`Vyk3/yt-tool`
- 相关 PR：`#26`、`#27`
- 记录日期：`2026-04-19`

## 事实时间线

1. 创建 `PR #26` 时使用 `gh pr create --body "<多行内容+代码块>"`，shell 对正文中的片段发生解释，出现 `command not found` 噪音。
2. 后续通过 `gh pr edit --body ...` 修正了正文，但流程上属于“先污染、后修补”。
3. 合并阶段使用 `gh pr merge --squash --delete-branch`，触发本地 worktree 分支占用冲突（`main is already used by worktree`）。
4. 之后切换为“先 merge，不在同一步删除分支”的路径，完成合并。
5. 在被用户指出前，未第一时间执行结构化复盘，导致规则沉淀滞后。

## 根因

### 流程缺口

- 缺少“PR body 一律文件输入”的前置检查，仍允许临场拼接多行参数。
- 多 worktree 场景下，未把“merge 与删分支分离”作为默认策略。
- 命令异常后没有立即触发最小复盘收口。

### 技术缺口

- 低估了 shell 对多行 body 的解析风险（尤其包含代码块/特殊字符时）。
- 误判 `gh pr merge --delete-branch` 在本地 worktree 占用分支时的行为。

## 临时止血动作

- 统一改用 `--body-file` 更新 PR 正文。
- merge 时不再携带 `--delete-branch`，先完成合并，再单独清理分支。
- 对 `PR #27` 重建干净分支并强制更新，收敛 diff 范围后再跑全量 checks。

## 永久防护
