# Allow Candidate Governance

本文件定义 allow 候选的分级与准入规则，目标是让“哪些命令可以被预先授权”有稳定标准，而不是会话内临时拍板。

本规则同时覆盖两类运行时入口：

- Codex 的 prefix rules，例如本地 `default.rules`
- Claude Code 的项目级 `permissions.allow`，例如 `.claude/settings.local.json`

如果只治理文档，不治理实际运行时权限配置，规则会与真实执行边界脱节。

## 设计目标

- 降低高频低风险命令的重复确认成本
- 保持操作边界清晰，可审计、可回放
- 避免 allow 集合无限膨胀

allow 规则集采用保守扩张策略。新增候选只可来自复盘、重复执行记录或明确的高频工作流。

## 分级

### A 类：纯只读查询

特征：

- 不修改工作树
- 不修改远端状态
- 失败后无残留副作用

典型命令：

- `git status --short`
- `git diff --stat`
- `git branch --show-current`
- `rg`
- `rg --files`
- 只读状态查询脚本
- 只读测试 / 检查命令

默认策略：最适合作为 allow 候选。

### B 类：低风险本地操作

特征：

- 只影响本地仓库或本地工作目录
- 风险可控且易回溯
- 参数边界稳定

典型命令：

- `git add`
- `git checkout -b`
- 仓库内固定验证命令
- 项目内固定格式整理脚本

默认策略：可进入 allow，但必须限定前缀和参数模式。

### C 类：远程可见但可恢复操作

特征：

- 会影响远端或第三方系统可见状态
- 理论上可撤销或修正，但成本更高

典型命令：

- `git push`
- PR 创建
- 远程 CI 轮询与状态查询

默认策略：不自动 allow。仅在明确需要时以更窄的前缀单独准入。

### D 类：不可逆或高风险操作

特征：

- 可能删除数据、改写历史、覆盖文件或发布外部产物
- 失败后代价高，回退成本不可忽略

典型命令：

- `git reset --hard`
- `rm`
- release 发布
- 强制推送
- 覆盖写敏感文件

默认策略：始终逐次确认，不进入快捷 allow。

## 准入标准

allow 候选必须同时满足：

- **高频**：同类任务中会反复执行
- **参数边界清晰**：能被稳定约束为固定前缀或固定模板
- **风险低且可预期**：不会引入隐蔽副作用
- **失败可承受**：失败不会破坏工作树或远端状态
- **可通过前缀规则稳定授权**：不依赖复杂 shell 特性

不满足任何一条，都不应进入长期 allow。

## 拒绝条件

以下命令即使高频，也不进入快捷 allow：

- 带通配符的命令
- 带重定向的命令
- 删除命令
- 覆盖写命令
- 任意脚本解释器裸调用，例如不受约束的 `python -c`、`bash -c`
- 依赖 heredoc、命令替换、复杂管道的命令

原因不是“看起来危险”，而是参数空间过大，无法稳定审计。

## 候选产生方式

每轮会话结束后可以产出 allow 候选建议，但该建议只是一份评估，不自动生效。

建议至少包含：

- 命令或前缀
- 所属运行时（Codex / Claude）
- 所属分级
- 高频证据
- 预期收益
- 风险边界
- 推荐是否入库

## 入库要求

候选正式入库前，必须完成：

1. 分类
2. 准入标准核对
3. 拒绝条件排除
4. 人工确认
5. 必要的前缀收缩

如果只能通过放宽前缀才能覆盖需求，则优先不入库。

## 现有规则集的保留与待确认

评估现有 allow 规则集时，不只看“是否曾经有用”，还要看它是否仍然满足当前准入标准。

### 建议继续保留

满足以下条件时，建议继续保留：

- 仍然高频使用
- 前缀边界清晰，没有逐步扩权
- 对应命令仍与当前仓库工作流一致
- 不会绕过本仓库现有的安全约束

典型例子：

- 只读查询命令
- 固定验证命令
- 已收缩到仓库脚本或虚拟环境入口的验证前缀
- 已明确绑定仓库、分支或文件路径的低风险 Git 命令

### 建议转为待确认

出现以下任一情况时，规则应转为“待用户确认”而不是继续默认保留：

- 长时间未使用，收益不明确
- 前缀过宽，存在被顺带扩权的风险
- 对应工作流已经变化，旧命令不再是推荐入口
- 涉及远程可见操作，但历史上只是为某一次任务临时放开
- 路径、仓库、分支或环境绑定已经失效

### 建议移除

出现以下情况时，应从长期规则集中移除：

- 命令已废弃或被新脚本入口替代
- 继续保留会制造错误暗示，例如“它仍被推荐使用”
- 维护者已经无法解释该前缀的现实收益和边界

## 定期评估与修剪

allow 规则集需要定期评估。否则它只会单向膨胀，逐步偏离“高频、低风险、边界清晰”的初衷。

建议在以下时机触发 review：

- 每次发布前
- 同一仓库累计新增 3 条以上 allow 后
- 某条 allow 连续两个迭代未使用
- 工作流、脚本入口或 CI 策略发生变化后

### 修剪流程

1. 列出当前 allow 规则
2. 按 A / B / C / D 类重新分类
3. 检查是否仍满足准入标准
4. 标记为 `保留` / `待确认` / `移除`
5. 对 `待确认` 项在下一次相关任务中向用户确认
6. 对 `移除` 项删除并记录原因

### 最小评估记录

每条被 review 的 allow 至少记录：

- 所属运行时
- 当前用途
- 最近使用场景
- 当前分级
- 是否仍满足准入标准
- 处理结果：`保留` / `待确认` / `移除`

如果没有最近使用场景或无法说明边界，默认不应继续长期保留。

## Hook 与权限过滤

Claude 的 hook 不直接等价于 allow 规则，但它们会改变运行时行为，也应纳入同一次治理评估。

重点检查：

- 是否仍符合当前输出裁剪协议
- 是否会掩盖关键失败信息
- 是否与文档中声明的展示预算冲突

因此，运行时治理的最小盘点对象应包括：

- Codex prefix rules
- Claude `permissions.allow`
- Claude hooks
