# AGENTS.md

## 目标

本文件定义本项目中 agent 的长期工作流原则与约束。
目标是提升长任务中的稳定性、可验证性、可维护性与交付一致性。

本文件只放“原则”，不放单次任务临时步骤。
单次任务要求应放在 issue、PR 描述或独立 checklist 文件中。

---

## 规则源与优先级

### 规则源
- 仅项目根目录 `AGENTS.md` 作为规范源。
- 不在子目录维护第二份 `AGENTS.md` / `agents.md`。
- 若发现多份冲突，先对齐到项目根版本，再继续执行。

### 指令优先级
按以下顺序执行：
1. 系统/平台安全策略
2. 开发者指令
3. 用户当前明确要求
4. 本文件（`AGENTS.md`）

说明：
- 用户可覆盖流程（如“只 review”/“跳过 review”/“不要跑测试”），但需在结果中标注风险。
- 若用户要求与更高优先级冲突，遵循更高优先级并明确说明原因。

## 模型分工

### 原则：能力角色优先，型号映射其次
- 先按任务能力选择角色，再按可用性选择模型。
- 型号升级/下线时，优先更新“映射关系”，不改流程原则。

### 能力角色
- `Planner`：任务澄清、范围定义、风险分析、阶段拆分、验收标准。
- `Scout`：代码检索、摘要、依赖梳理、小补丁和测试草案。
- `Builder`：主实现、联调修复、测试补齐与交付推进。
- `Reviewer`：独立复核、回归风险判断、上线前把关。

### 当前默认映射（可变）
- `Planner`：GPT-5.4
- `Scout`：GPT-5.4 mini
- `Builder`：GPT-5.3 Codex
- `Reviewer`：优先 GPT-5.4（可由能力等价模型替代）

### 可用性降级
- 若指定模型不可用，可用能力等价模型替代。
- 替代时必须在结果中注明“原计划模型/实际模型/影响”。

### 模型版本更新同步流程
1. 仅更新本节映射关系，不改其他流程段落。
2. 映射更新后至少执行一次小任务和一次中任务回归。
3. 新映射首轮使用时记录“旧映射/新映射/影响”。
4. 若新映射不稳定，临时回退上一版并记录原因。

### 任务路由（先评估再选流程）
- 进入执行前，必须先评估任务画像：
  - 任务类型（修 bug / 新功能 / 文档 / 发布 / CI / 迁移）
  - 改动范围（文件数、模块边界、是否跨层）
  - 风险级别（接口兼容、并发状态、安全、发布链路、数据/配置）
  - 验证成本（是否可快速自动化验证、是否依赖真实环境）
- 规模判定（用于单/多 agent 决策）：
  - 小：通常 <=3 文件、单模块、无公共接口或发布链路变更
  - 中：通常 4-8 文件或跨 2 个模块，需要联动验证
  - 大：通常 >8 文件、跨层改动、含迁移/发布/跨平台链路
- 路由规则：
  - 满足“小范围 + 低风险 + 验证路径清晰”时，可走单 agent 本地推进。
  - 中大型、跨模块、高风险、长链路或可并行拆分任务，默认采用多 agent 协作。
- 强制多 agent 触发器（任一满足即触发）：
  - 发布流程、打包链路、CI 基础设施变更
  - 对外接口/公共契约变更
  - 跨平台行为修复（涉及至少 2 个 OS）
  - 同一问题连续 2 轮以上回归修复
- 用户明确指定协作方式时，以用户要求为准；若与高优先级规则冲突，需明确说明并降级处理。

## 默认流程

### 标准六步（推荐用于中大型 / 高风险任务）
1. 用 GPT-5.4 输出计划：目标、范围、不做什么、风险、验收标准。
2. 生成阶段合同：当前阶段目标、完成定义、验证方法、兼容性约束。
3. 将局部准备工作下发给 GPT-5.4 mini（检索/摘要/依赖梳理/小补丁）。
4. 由 GPT-5.3 Codex 执行主实现。
5. 由独立评估角色复核阶段结果。
6. 交回 GPT-5.4 做合并前复核。

### 小任务快速通道（仅在路由评估满足条件时）
满足以下条件可走简化流程（实现 -> 验证 -> 简审）：
- 单文件或小范围改动
- 低风险
- 回滚成本低
- 验证路径清晰

## 阶段准入与退出

### 阶段准入（DoR）
进入实现前必须明确：
- 目标
- 范围
- 不做什么
- 验收标准
- 验证方式

### 阶段完成（DoD）
阶段结束必须具备：
- 代码或文档产出
- 可复现验证结果
- 风险记录
- 下一步建议

## Harness 约束增强（原则级）

### 中高风险任务强制评估 Gate
- 中/大规模或高风险任务必须经过独立 `Reviewer` gate，未通过不得进入阶段完成。
- Gate 采用“混合阈值”：
  - 硬门槛：`功能正确性`、`安全性`、`兼容性`。任一不达标，阶段不通过。
  - 建议分：`可维护性`、`交互体验`。未达标可在风险可控前提下有条件放行，但必须记录技术债与后续计划。
- 小任务快速通道保持简化流程，不因中高风险 gate 机制被无差别加重。

### 阶段合同协商闭环
- 实现前，`Builder` 必须先提交“可验证阶段合同”（本阶段目标、完成定义、验证证据）。
- `Reviewer` 或等价独立评估角色确认合同后，方可进入实现。
- 执行中若范围、约束或验收口径发生变化，必须先更新合同再继续推进。

### 评估证据要求
- 评估不得仅基于 `git diff` 或静态阅读结论，必须覆盖至少一类真实可运行路径证据：
  - API / 状态 / 实际运行 / 用户路径
- 对“表面完成、实为 stub 或占位实现”的情况应显式拦截，不得按已完成放行。

### 上下文策略
- 默认可采用连续会话与上下文压缩。
- 若出现长任务漂移、提前收尾倾向、关键信息遗失等迹象，应升级为 `context reset + structured handoff`。
- 结构化交接至少包含：目标、已完成、未完成、当前风险、关键约束、改动文件清单。

### 负载复盘与预算止损
- 模型升级或关键映射变更后，按“一次只移除一个 harness 组件”的方式进行回归复盘，识别真实承重组件。
- 中长任务应在开始前设定预算上限（迭代轮次 / 时间 / 成本中的至少一项）。
- 触发预算止损时，必须记录：超限原因、已验证收益、下一步最小实验。

## 交接原则

agent 之间通过文件交接，不依赖隐式聊天上下文。

推荐文件：
- `AGENTS.md`
- `plan.md`
- `contract.md`
- `review.md`
- `handoff.md`

交接最少应包含：
- 已完成内容
- 未完成内容
- 当前风险
- 下一步建议
- 关键约束
- 改动文件清单

并行协作约束：
- 一文件一 owner，避免并行改同一文件
- 未经确认不回滚他人改动

## 评估原则

- 优先由独立评估角色进行最终复核。
- 若受资源或上下文限制无法获得独立评估，可降级为实现者自评，但必须在结果中注明：
  - 降级原因
  - 未覆盖风险
- 评估至少覆盖：
  - 功能完整性
  - 需求一致性
  - 交互正确性
  - 代码质量
  - 回归风险

任何关键维度不达标，阶段不通过。

---

## 验证原则

验证应先低成本收敛，再补高价值真实路径。

默认验证顺序：
1. 自动化测试
2. 静态检查
3. API 与状态检查
4. 实际运行
5. 用户路径操作

补充规则：
- 涉及交互、发布链路、跨平台差异或高风险改动时，必须覆盖 4/5 中至少一项真实路径验证。
- 若真实路径验证无法执行，按降级模式记录原因与风险，不得省略说明。

验证命令优先来源：
1. 本文件约定
2. 项目配置（如 `.Codex/settings.json`）
3. `package.json` scripts / `Makefile`
4. 项目现有脚本

本仓库默认：

```yaml
lint: ruff check app/ tests/
test: python -m pytest tests/ -q
```

验证结论必须分类：
- `Passed`
- `Failed`
- `Not Run`
- `Unavailable`
- `Not Configured`

### CI / 跨平台稳健性原则
- 新增 smoke 或回归步骤时，依赖安装应“按场景最小化”：
  - CLI smoke 只装 CLI 必需依赖，不引入 GUI 链路依赖。
  - GUI 依赖（如 `pywebview`）仅在需要 GUI 行为验证的步骤安装。
- Windows runner 中不得假设 `py -3` 会自动命中目标版本：
  - 需要明确固定解释器来源（如 setup-python 的 `python`，或显式约束 `PY_PYTHON`）。
- 跨平台日志/提示若包含非 ASCII 文本（如中文、框线字符），验证步骤需显式考虑编码环境（如 UTF-8）。
- 预期“非零退出码即成功语义”的 smoke（例如故意不给 URL）必须在脚本内显式归一化：
  - 先捕获并校验退出码，再将步骤最终退出设为成功。
  - 不依赖 shell 默认行为隐式判定通过。
- 若某模块在无可选依赖环境中也需要被测试导入，应避免顶层强依赖导入（可采用延迟导入或可选分支）。

### CI 失败止损原则
- 同一流水线若连续失败 2 次，先暂停继续推送，必须先完成失败归因：
  - 失败步骤与第一条关键报错
  - 首个失败提交范围
  - 最可能根因与验证计划
- 每轮 CI 修复尽量“一次只改一个失败模式”，避免多变量叠加导致难以归因。
- 无法在本地完全复现 CI 时，必须在结果中注明“不可复现部分”与“下一步最小实验”。

### 主分支同步安全原则
- 同步 `main` 前，若工作区非干净，先保护本地改动（含 untracked）再执行 fast-forward。
- 当 `pull --ff-only` 被“本地修改/未跟踪文件将被覆盖”阻塞时：
  - 先处理冲突文件（备份、移动或 stash），禁止直接覆盖。
  - 同步完成后恢复本地改动，并再次检查冲突状态。
- 清理动作优先“可逆”：
  - 临时文件优先重命名/备份再删除。
  - 仅在确认恢复完成后再清理临时 stash。

### 分支与提交流程原则
- 默认在非 `main` 分支完成开发、验证与 review，再合并到 `main`。
- 仅在以下情况可直接在 `main` 推进：
  - 用户明确要求
  - 极小且低风险的紧急修复，且可立即完成验证与回滚
- 直接在 `main` 推进时，必须在结果中说明原因与风险。

## 复杂度控制

- 默认采用最简单可行结构。
- 仅在必要时增加 planner/evaluator/多轮 QA/更细阶段拆分。
- 新增任何 harness 组件时，必须说明其解决的失败模式。
- 定期复盘 scaffold：保留承重部分，移除冗余开销。

## Codex 权限与安全执行原则

- 本节仅定义 Codex 工作流原则，不与其他工具工作流互相引用或互为依赖。
- 项目内权限原则不等于可执行白名单；实际执行权限由平台沙箱与用户批准规则共同决定。

### 操作分级
- 低风险操作（只读查询、无远程副作用、无破坏性）可直接执行。
- 中风险操作（会修改工作区但可回滚）必须满足最小改动面、可验证、可追溯三项约束。
- 高风险操作（远程可见、不可逆、破坏性）必须先获得用户明确确认，再执行。

### 白名单申请约束
- 申请放行规则时仅允许“最小可用前缀”，禁止过宽规则。
- 不得通过白名单或前缀规则绕过高优先级安全策略。
- 若存在更小授权范围可满足目标，应优先选择更小范围方案。

### 受限披露要求
- 因权限、沙箱或环境限制导致操作未完成时，必须在结果中明确写出：
  - 受限环节
  - 影响范围
  - 未覆盖风险

## Fallback（降级）

以下情况进入降级模式：
- 不是 git 仓库
- 没有可审查 diff
- 关键验证无法执行
- 环境受限导致目标路径不可验证

降级要求：
- 仅降级受影响环节
- 明确写出降级原因、影响范围、未覆盖风险
- 不将降级结果表述为“完整通过”

## 最终输出模板

最终报告至少包含：
- `Review 发现`
- `已修复`
- `阶段合同结果`
- `评估门槛判定`
- `验证结果`
- `预算与止损记录`
- `降级说明（若有）`
- `遗留风险`

## 最终原则

优先正确性、可维护性、可验证性、安全性，以及最小改动面。
