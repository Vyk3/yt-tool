# AGENTS.md

## 目标

本文件定义本项目中 agent 的长期工作流原则与约束。
目标是提升长任务中的稳定性、可验证性、可维护性与交付一致性。

本文件只放“原则”，不放单次任务临时步骤。
单次任务要求应放在 issue、PR 描述或独立 checklist 文件中。

---

## 规则源与优先级

### 规则源
- 仅项目根目录 `AGENTS.md` 作为规范源。
- 不在子目录维护第二份 `AGENTS.md` / `agents.md`。
- 若发现多份冲突，先对齐到项目根版本，再继续执行。

### 指令优先级
按以下顺序执行：
1. 系统/平台安全策略
2. 开发者指令
3. 用户当前明确要求
4. 本文件（`AGENTS.md`）

说明：
- 用户可覆盖流程（如“只 review”/“跳过 review”/“不要跑测试”），但需在结果中标注风险。
- 若用户要求与更高优先级冲突，遵循更高优先级并明确说明原因。

---

## 模型分工

### GPT-5.4
负责：
- 任务澄清
- 目标与范围定义
- 风险分析
- 阶段拆分
- 验收标准制定
- 合并前复核与最终裁决

适用：
- 模糊需求
- 高风险修改
- 跨模块任务
- 架构、鉴权、数据、部署相关任务

### GPT-5.4 mini
负责：
- 代码检索
- 大文件摘要
- 局部代码修改
- 文档整理
- 小型测试补充
- 并行子任务处理

适用：
- 边界清晰
- 低风险
- 局部、重复、可快速验证任务

### GPT-5.3 Codex
负责：
- 多文件实现
- 长链路 feature 开发
- 仓库内持续施工
- 代码审查
- 返工修复与测试完善

适用：
- 目标和约束已明确后的主实现阶段
- 需要多轮编码推进的中大型改动

### 可用性降级
- 若指定模型不可用，可用能力等价模型替代。
- 替代时必须在结果中注明“原计划模型/实际模型/影响”。

---

## 默认流程

### 标准六步
1. 用 GPT-5.4 输出计划：目标、范围、不做什么、风险、验收标准。
2. 生成阶段合同：当前阶段目标、完成定义、验证方法、兼容性约束。
3. 将局部准备工作下发给 GPT-5.4 mini（检索/摘要/依赖梳理/小补丁）。
4. 由 GPT-5.3 Codex 执行主实现。
5. 由独立评估角色复核阶段结果。
6. 交回 GPT-5.4 做合并前复核。

### 小任务快速通道
满足以下条件可走简化流程（实现 -> 验证 -> 简审）：
- 单文件或小范围改动
- 低风险
- 回滚成本低
- 验证路径清晰

---

## 阶段准入与退出

### 阶段准入（DoR）
进入实现前必须明确：
- 目标
- 范围
- 不做什么
- 验收标准
- 验证方式

### 阶段完成（DoD）
阶段结束必须具备：
- 代码或文档产出
- 可复现验证结果
- 风险记录
- 下一步建议

---

## 交接原则

agent 之间通过文件交接，不依赖隐式聊天上下文。

推荐文件：
- `AGENTS.md`
- `plan.md`
- `contract.md`
- `review.md`
- `handoff.md`

交接最少应包含：
- 已完成内容
- 未完成内容
- 当前风险
- 下一步建议
- 关键约束
- 改动文件清单

并行协作约束：
- 一文件一 owner，避免并行改同一文件
- 未经确认不回滚他人改动

---

## 评估原则

- 实现者不能作为最终评估者。
- 评估至少覆盖：
  - 功能完整性
  - 需求一致性
  - 交互正确性
  - 代码质量
  - 回归风险

任何关键维度不达标，阶段不通过。

---

## 验证原则

真实行为验证优先于静态阅读。

验证优先顺序：
1. 实际运行
2. 用户路径操作
3. API 与状态检查
4. 自动化测试
5. 静态检查

验证命令优先来源：
1. 本文件约定
2. 项目配置（如 `.Codex/settings.json`）
3. `package.json` scripts / `Makefile`
4. 项目现有脚本

本仓库默认：

```yaml
lint: ruff check app/ tests/
test: python -m pytest tests/ -q
```

验证结论必须分类：
- `Passed`
- `Failed`
- `Not Run`
- `Unavailable`
- `Not Configured`

---

## 复杂度控制

- 默认采用最简单可行结构。
- 仅在必要时增加 planner/evaluator/多轮 QA/更细阶段拆分。
- 新增任何 harness 组件时，必须说明其解决的失败模式。
- 定期复盘 scaffold：保留承重部分，移除冗余开销。

---

## Fallback（降级）

以下情况进入降级模式：
- 不是 git 仓库
- 没有可审查 diff
- 关键验证无法执行
- 环境受限导致目标路径不可验证

降级要求：
- 仅降级受影响环节
- 明确写出降级原因、影响范围、未覆盖风险
- 不将降级结果表述为“完整通过”

---

## 最终输出模板

最终报告至少包含：
- `Review 发现`
- `已修复`
- `验证结果`
- `遗留风险`

---

## 最终原则

优先正确性、可维护性、可验证性、安全性，以及最小改动面。
